# Security policy

## Reporting a concern

Please do not open a public issue for a vulnerability that could expose credentials, private data, repository write access, or a live third-party system.

Send a private report through GitHub's private vulnerability reporting feature when available. Include:

- affected file, workflow, or research artifact
- a concise description of the issue
- conditions required to reproduce it
- likely impact
- a safe reproduction using synthetic data
- suggested mitigation, when known

Do not include real secrets, private user information, harmful payloads, or access to systems you do not own.

## Scope

Reports about the repository's code, validation workflow, published test harnesses, and accidental data exposure are in scope.

Disagreement with a research conclusion is not a vulnerability. Open a normal issue or pull request for evidence corrections, taxonomy disputes, missing limitations, or source updates.

## Research conduct

Testing must be limited to systems you own or are explicitly authorized to assess. Findings involving a vendor or third party should follow that organization's disclosure process before public discussion.

## Response goals

The maintainer will acknowledge a credible private report, assess whether public material must be removed or corrected, and document the eventual fix without exposing sensitive reproduction details.
